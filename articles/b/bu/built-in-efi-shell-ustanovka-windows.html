Установка Windows через встроенную EFI Shell: полное руководство

  • Навигация по файловой системе EFI Shell
  • Запуск установщика Windows вручную

Попытка установить операционную систему на современный компьютер или сервер часто приводит к неожиданному препятствию: вместо привычного меню выбора языка или логотипа установщика на экране появляется черный фон с белым текстом и приглашением Shell>.

Эта ситуация возникает, когда прошивка UEFI не может найти загрузочный файл в стандартном расположении или когда структура разделов на накопителе не соответствует ожиданиям системы безопасности Secure Boot.

Пользователь оказывается один на один с интерфейсом командной строки, где отсутствуют графические подсказки, а стандартные методы установки через BIOS перестают работать, требуя ручного вмешательства в процесс инициализации загрузчика.

Диагностика среды и подготовка загрузочного носителя

Прежде чем приступать к ручному запуску инсталлятора, необходимо убедиться, что используемый установочный носитель корректно подготовлен для работы в режиме UEFI, так как legacy-режим (BIOS) в данной среде часто недоступен или отключен политиками безопасности.

На практике создание загрузочной флешки требует использования специализированного софта, который правильно запишет таблицу разделов GPT и разместит файлы EFI в нужной директории, иначе система просто проигнорирует устройство при старте.

Если вы видите сообщение No Boot Device Found перед переходом в Shell, это верный признак того, что загрузочная запись отсутствует или повреждена, и повторное форматирование носителя с правильными параметрами является обязательным шагом.

Важно проверить, включен ли режим UEFI Boot в настройках прошивки материнской платы, поскольку встроенная оболочка EFI Shell часто активируется автоматически при отсутствии других приоритетных устройств загрузки.

Создание совместимого загрузочного накопителя

Для успешного запуска инсталляции необходимо использовать утилиту Rufus или аналогичный инструмент, выбрав схему раздела GPT и целевую систему UEFI (non CSM), что гарантирует создание правильной структуры папок.

Критически важным является наличие папки EFI/BOOT в корне флешки, внутри которой должен находиться файл BOOTX64.EFI, так как именно его ищет прошивка при автоматическом сканировании устройств.

Если стандартные образы ISO не содержат правильной структуры, может потребоваться ручное копирование файлов или использование командной строки для воссоздания загрузочного сектора, что особенно актуально для корпоративных сборок Windows.

⚠️

Неправильная схема разделов (MBR вместо GPT) приведет к невозможности установки Windows в режиме UEFI без полной переразметки диска с потерей данных.

Стоит учитывать, что некоторые материнские платы требуют отключения функции Secure Boot для запуска не сертифицированных загрузчиков, хотя официальные образы Microsoft обычно проходят проверку подписи без проблем.

После записи образа рекомендуется проверить содержимое флешки через проводник, убедившись, что файлы видны и структура папок соответствует требованиям спецификации UEFI для избежания ошибок на этапе старта.

Навигация по файловой системе EFI Shell

После попадания в среду EFI Shell пользователь видит командную строку, которая позволяет взаимодействовать с файловой системой накопителей на низком уровне, минуя стандартные механизмы загрузки ОС.

Первым делом необходимо определить, какой буквой дискового устройства обозначена ваша флешка, так как система может присвоить ей неочевидное обозначение, отличное от привычного FS0 или FS1.

Для просмотра списка доступных устройств и файловых систем используется команда map -r, которая пересканирует шины и выведет актуальную карту подключенных накопителей, включая USB-порты и внутренние диски.

В выводе команды можно увидеть устройства с префиксом BLK (блокирующие устройства) и FS (файловые системы), и нас интересуют именно последние, так как они содержат файловую структуру FAT32.

Обозначение Тип устройства Описание
FS0, FS1 File System Разделы FAT32, доступные для чтения (обычно флешки или EFI-разделы)
BLK0, BLK1 Block Device Физические блоки устройств, требуют монтирования для доступа к файлам
PcieRoot Path Путь к устройству в иерархии ACPI, используется для идентификации

Определение буквы диска и переход к файлам

Для перехода на нужное устройство достаточно ввести его обозначение с двоеточием, например, FS1:, после чего приглашение командной строки изменится, подтверждая смену текущего контекста работы.

Если при вводе команды dir или ls отображаются файлы установщика Windows, такие как setup.exe и папка EFI, значит, вы находитесь на правильном носителе и готовы к запуску.

В случае, когда на устройстве ничего не видно, стоит попробовать другую букву из списка map или проверить, отформатирован ли раздел в файловой системе FAT32, которую нативно поддерживает EFI.

Навигация по папкам осуществляется стандартными командами cd (change directory), что позволяет спуститься в каталог EFI/BOOT для ручного поиска исполняемого файла загрузчика.

📝

Команда map -r обязательна к выполнению каждый раз после подключения новой флешки, так как EFI Shell не всегда определяет горячее подключение автоматически.

Запуск установщика Windows вручную

Основная задача на данном этапе — найти и запустить исполняемый файл загрузчика, который инициирует цепочку загрузки установочного окружения Windows PE.

Стандартным путем является переход в директорию EFI/BOOT и запуск файла BOOTX64.EFI, однако в некоторых случаях требуется прямой запуск setup.exe из корня или папки Sources.

Если автоматический запуск не происходит, можно использовать команду fs0:\EFI\Boot\bootx64.efi, указав полный путь к файлу, что гарантирует запуск нужного исполняемого модуля независимо от текущей директории.

После успешного запуска файла экран может на несколько секунд погаснуть или моргнуть, что свидетельствует о передаче управления коду загрузчика и начале инициализации графического интерфейса установки.

Альтернативные методы инициализации загрузки

В ситуациях, когда стандартный путь не работает, можно попробовать запустить файл shellx64.efi из папки EFI/Microsoft/Boot, если он присутствует на носителе, что иногда помогает обойти ошибки проверки подписи.

Для серверных платформ или специфических сборок может потребоваться указание дополнительных параметров запуска, хотя для стандартной установки Windows 10/11 это требуется крайне редко.

Если после запуска файла снова появляется оболочка Shell, это означает, что исполняемый файл поврежден, несовместим с архитектурой процессора или заблокирован настройками безопасности UEFI.

cd \EFI\Boot


bootx64.efi

Стоит помнить, что запуск через EFI Shell часто является временным решением, и после первой успешной загрузки установщика рекомендуется проверить настройки приоритета загрузки в BIOS для будущих перезагрузок.

Решение проблем с разметкой диска и GPT

Частой причиной невозможности установки является несовместимость стиля разделов диска, когда система требует GPT, а диск размечен в MBR, или наоборот, что приводит к ошибке на этапе выбора раздела.

Встроенная EFI Shell сама по себе не предназначена для конвертации разделов, но она позволяет запустить установщик, который, в свою очередь, может иметь ограничения на запись данных.

Если установщик Windows сообщает, что выбранный диск имеет стиль разделов MBR, необходимо либо конвертировать диск, либо изменить настройки BIOS, хотя для UEFI предпочтителен именно стиль GPT.

Для конвертации диска можно использовать встроенную утилиту diskpart, доступную через комбинацию клавиш Shift+F10 в окне установки, что позволяет очистить диск и создать новую структуру разделов.

Параметр MBR (Legacy) GPT (UEFI)
Макс. размер диска 2 ТБ 9.4 Зеттабайт
Кол-во разделов 4 основных 128 и более
Загрузочный сектор MBR EFI System Partition

Использование Diskpart для подготовки диска

После запуска установщика и появления экрана выбора языка или раздела, нажмите Shift+F10 для вызова командной строки, где можно выполнить очистку диска командой clean.

Затем выполните команду convert gpt, чтобы изменить стиль разделов на современный стандарт, необходимый для загрузки в режиме UEFI без ошибок совместимости.

После завершения конвертации закройте окно командной строки и нажмите кнопку обновления списка дисков, чтобы увидеть доступное для установки пространство.

💡

Команда clean удаляет все данные на выбранном диске, поэтому убедитесь, что выбрали правильное устройство, особенно если в системе несколько накопителей.

Типичные ошибки и методы их устранения

В процессе ручной установки через EFI Shell пользователи часто сталкиваются с повторяющимися проблемами, связанными с путями к файлам, правами доступа или аппаратной совместимостью.

Понимание природы этих ошибок позволяет быстро диагностировать проблему и применить правильное решение, не прибегая к переустановке системы или замене оборудования.

Ниже приведен список наиболее распространенных ситуаций и алгоритмы действий для их разрешения.

  1. Ошибка «Selected boot image did not authenticate» — это сообщение указывает на то, что функция Secure Boot блокирует запуск файла, так как его цифровая подпись не распознана; для решения необходимо временно отключить Secure Boot в настройках BIOS или использовать сертифицированный образ.
  2. Команда «map» не показывает флешку — часто возникает из-за того, что флешка отформатирована в NTFS, который EFI Shell не видит без дополнительных драйверов; решение заключается в переформатировании накопителя в FAT32.
  3. Запуск файла возвращает в Shell — означает, что исполняемый файл предназначен для другой архитектуры (например, IA32 вместо X64) или поврежден; следует перепроверить образ ISO и заново записать его на носитель.
  4. Отсутствует файл bootx64.efi — в некоторых дистрибутивах файл может называться иначе или находиться в другой папке; используйте команду find -n boot*.efi для поиска исполняемых файлов EFI на диске.
📌

Успешная установка через EFI Shell зависит от правильной структуры FAT32, наличия файла bootx64.efi и отключения конфликтующих настроек безопасности.

Завершая процесс, стоит отметить, что ручная установка через встроенную оболочку EFI Shell является мощным инструментом в арсенале системного администратора.

Этот метод позволяет обойти ограничения стандартного загрузчика, исправить ошибки разметки и установить операционную систему даже на системах с нестандартной конфигурацией BIOS.

После успешной установки Windows рекомендуется вернуться в настройки BIOS и вернуть приоритет загрузки на стандартный Windows Boot Manager, чтобы система загружалась автоматически без необходимости ручного вмешательства.
