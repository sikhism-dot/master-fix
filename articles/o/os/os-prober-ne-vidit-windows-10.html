os-prober не видит Windows 10: решение проблемы

  • Настройка UEFI и Secure Boot
  • Проблема быстрой загрузки Windows

Ситуация, когда после установки дистрибутива Linux система GRUB не отображает Windows 10 в меню загрузки, является одной из самых распространенных проблем при настройке двойной загрузки. Пользователь сталкивается с тем, что компьютер загружает только что установленную ОС, игнорируя существующую систему Microsoft, хотя файлы Windows физически находятся на жестком диске.

Чаще всего причина кроется в изменении политики безопасности утилиты os-prober, которая по умолчанию перестала сканировать другие разделы в новых версиях GRUB. Это сделано для предотвращения потенциальных конфликтов и повышения безопасности, однако для конечного пользователя это означает необходимость ручной правки конфигурационных файлов.

Отсутствие записи о Windows в меню загрузчика не означает повреждение файлов операционной системы. Обычно достаточно внести несколько изменений в конфигурационный файл /etc/default/grub и принудительно запустить обновление загрузчика, чтобы восстановить доступ к обоим системам.

Диагностика и проверка разделов

Первым шагом перед внесением любых изменений в конфигурацию загрузчика является убедительное подтверждение того, что раздел с операционной системой Windows действительно существует и доступен для чтения. Если диск отформатирован в файловую систему NTFS, Linux сможет его идентифицировать, но для работы os-prober раздел должен быть смонтирован или хотя бы видим в таблице разделов.

Для начала стоит воспользоваться утилитой lsblk, которая выведет список всех блочных устройств. В выводе команды необходимо искать разделы с типом файловой системы ntfs или ntfs-3g. Если такой раздел присутствует, это подтверждает, что Windows никуда не пропала и проблема носит программный характер.

Допустим, команда lsblk показала наличие раздела /dev/sda1 с файловой системой NTFS. В этом случае можно утверждать, что данные целы. Однако, если раздел не отображается вообще, возможно, требуется проверить физическое подключение диска или состояние RAID-контроллера в BIOS/UEFI.

📝

В некоторых дистрибутивах, таких как Ubuntu 22.04 и новее, утилита os-prober по умолчанию отключена и требует явного включения в конфигурации.

Далее следует выполнить непосредственное сканирование системы с помощью команды sudo os-prober. Если утилита работает корректно и разделы смонтированы, в терминале должна появиться строка, содержащая путь к загрузчику Windows, например /dev/sda1:Windows 10 (on /dev/sda1):efi:efi. Отсутствие вывода свидетельствует о том, что os-prober либо не может найти EFI-раздел, либо заблокирован настройками безопасности.

Проверка монтирования EFI-раздела

Критически важным моментом является корректное монтирование EFI-раздела, где хранятся загрузчики обеих систем. В современных системах с UEFI загрузчик Windows находится по пути /boot/efi/EFI/Microsoft или аналогичном, если раздел примонтирован. Если этот раздел не примонтирован в точку /boot/efi или /mnt/boot/efi, утилита os-prober может не найти загрузочные файлы.

Проверить статус монтирования можно командой mount | grep efi. Если вывод пуст, необходимо вручную примонтировать EFI-раздел перед запуском сканирования. Обычно это делается автоматически при установке системы, но в некоторых сценариях, особенно при ручной разметке диска, этот шаг мог быть пропущен.

Активация os-prober в конфигурации GRUB

Основной причиной игнорирования Windows является параметр GRUB_DISABLE_OS_PROBER, установленный в значение true или отсутствующий в новых версиях пакетов. Начиная с определенных обновлений GRUB, разработчики изменили поведение по умолчанию, требуя от пользователя явного согласия на сканирование других операционных систем.

Для исправления ситуации необходимо отредактировать основной файл конфигурации загрузчика. Откройте файл /etc/default/grub с правами суперпользователя, используя текстовый редактор, например nano или vim. Команда будет выглядеть как sudo nano /etc/default/grub.

В открывшемся файле следует найти строку, начинающуюся с GRUB_DISABLE_OS_PROBER. Если такая строка есть и установлена в true, измените значение на false. Если строки нет, добавьте новую строку в конец файла: GRUB_DISABLE_OS_PROBER=false. Это действие принудительно активирует механизм поиска других ОС.

📋 Активация os-prober

1Откройте терминал и введите sudo nano /etc/default/grub
2Найдите или добавьте строку GRUB_DISABLE_OS_PROBER=false
3Сохраните файл (Ctrl+O) и выйдите (Ctrl+X)
4Выполните команду обновления загрузчика

После внесения изменений в файл конфигурации недостаточно просто сохранить его. Необходимо перегенерировать файл grub.cfg, который непосредственно читается загрузчиком при старте системы. Для этого используется команда обновления, зависящая от дистрибутива. В Debian-подобных системах (Ubuntu, Mint, Debian) применяется sudo update-grub, которая является оберткой над grub-mkconfig.

В дистрибутивах на базе Arch Linux или Fedora команда может отличаться. Например, в Arch Linux используется sudo grub-mkconfig -o /boot/grub/grub.cfg. Важно использовать именно ту команду, которая соответствует вашей системе, чтобы изменения были применены корректно.

Нюансы для разных дистрибутивов

В дистрибутиве Ubuntu начиная с версии 22.04, параметр отключения добавлен по умолчанию, поэтому его изменение является обязательным шагом. В более старых версиях, таких как Ubuntu 20.04 или Linux Mint 20, сканирование могло работать автоматически, но после обновления пакетов GRUB настройки могли сброситься.

Пользователи Arch Linux часто сталкиваются с тем, что пакет os-prober необходимо установить отдельно, так как он не входит в базовую группу пакетов. Убедитесь, что пакет установлен командой pacman -S os-prober перед попыткой конфигурирования.

В системах Fedora и CentOS механизм работает аналогично, но пути к конфигурации могут незначительно отличаться в зависимости от версии загрузчика. Всегда проверяйте наличие файла /etc/default/grub перед редактированием.

Настройка UEFI и Secure Boot

Современные компьютеры используют интерфейс UEFI вместо устаревшего BIOS, что вносит свои коррективы в процесс загрузки. Одной из частых причин, по которой os-prober не видит Windows, является включенная функция Secure Boot. Эта функция безопасности блокирует загрузку не подписанных цифровыми сертификатами загрузчиков, к которым может относиться модифицированный GRUB.

Для решения проблемы необходимо войти в настройки BIOS/UEFI вашего компьютера. Обычно это делается нажатием клавиши F2, F12, Del или Esc сразу после включения питания. В меню настроек следует найти раздел, связанный с безопасностью (Security) или загрузкой (Boot).

⚠️

Отключение Secure Boot может снизить уровень защиты системы от вредоносного ПО. Делайте это только если уверены в источнике устанавливаемых ОС.

В найденном разделе необходимо найти опцию Secure Boot и переключить её в состояние Disabled. После этого сохраните настройки и перезагрузитесь. Также стоит проверить режим работы SATA контроллера: он должен быть установлен в AHCI, а не в RAID или IDE, так как в режиме RAID Linux может не видеть разделы Windows корректно.

Порядок загрузки в UEFI

Еще одним важным аспектом является приоритет загрузки. В настройках UEFI существует список Boot Option Priorities. Убедитесь, что загрузчик Linux (часто обозначаемый как ubuntu, fedora или просто grub) стоит первым в списке. Если первым стоит Windows Boot Manager, компьютер будет загружать Windows, минуя меню GRUB.

Если изменить порядок в BIOS невозможно, можно воспользоваться командой efibootmgr внутри Linux для управления порядком загрузки. Эта утилита позволяет изменять переменные EFI непосредственно из операционной системы.

Выполните команду sudo efibootmgr -v, чтобы увидеть текущий список загрузочных записей. Найдите номер записи, соответствующей GRUB, и используйте команду sudo efibootmgr -o 0001,0002,0003 (где цифры — номера записей), чтобы установить желаемый порядок.

Параметр Рекомендуемое значение Влияние
Secure Boot Disabled Разрешает загрузку сторонних загрузчиков
SATA Mode AHCI Обеспечивает видимость разделов в Linux
Fast Boot Disabled Предотвращает блокировку диска Windows

Проблема быстрой загрузки Windows

Функция «Быстрый запуск» (Fast Startup) в Windows 10 и 11 часто становится скрытой причиной проблем с двойной загрузкой. Когда эта функция активна, при выключении компьютер не выключается полностью, а переходит в состояние глубокого гибернации. В этом режиме диск Windows остается заблокированным для записи, и Linux не может получить к нему доступ или корректно считать загрузочную информацию.

Это приводит к тому, что os-prober либо не видит раздел, либо видит его как «грязный» и отказывается добавлять в меню загрузки во избежание потери данных. Решением является полное отключение быстрого запуска в настройках электропитания Windows.

Чтобы сделать это, загрузитесь в Windows, откройте Панель управления и перейдите в раздел Электропитание. Выберите действие «Действие кнопок питания» и нажмите «Изменение параметров, которые сейчас недоступны». В нижней части окна снимите галочку с пункта «Включить быстрый запуск».

💡

Если у вас нет доступа к Windows, можно попробовать отключить гибернацию командой powercfg /h off в командной строке с правами администратора.

После отключения функции необходимо выполнить полное выключение компьютера (не перезагрузку, а именно «Завершение работы») и только затем включать заново. Это гарантирует, что файловая система Windows будет закрыта корректно и доступна для чтения из Linux.

В некоторых случаях даже после отключения быстрого запуска раздел может оставаться заблокированным, если предыдущее завершение работы прошло некорректно. В такой ситуации может потребоваться однократная перезагрузка из Windows («Перезагрузка», а не «Выключение»), так как этот процесс всегда выполняет полное переподключение оборудования.

Типичные ошибки и методы их устранения

В процессе настройки двойной загрузки пользователи часто допускают ряд типичных ошибок, которые сводят на нет все усилия. Понимание этих ошибок позволяет быстрее диагностировать проблему и избежать потенциальной потери данных.

  1. Изменение порядка разделов — попытка переместить или изменить размер раздела Windows средствами Linux без создания резервной копии может привести к повреждению загрузочной записи и невозможности запуска Windows. Всегда делайте бэкап важных данных перед работой с разделами.
  2. Неверный целевой диск — установка загрузчика GRUB не на тот диск (например, на раздел с данными вместо EFI-раздела или диска с BIOS/UEFI) приводит к тому, что компьютер просто не видит меню загрузки. Убедитесь, что при установке Linux вы выбираете правильный диск для установки загрузчика.
  3. Игнорирование обновлений — использование старых версий os-prober или GRUB может содержать баги, исправленные в новых релизах. Перед настройкой стоит обновить пакеты системы до актуальных версий.
  4. Отсутствие проверки результата — выполнение команды обновления загрузчика без последующей проверки вывода команды os-prober или файла конфигурации. Всегда убеждайтесь, что в выводе команды обновления появилась строка «Found Windows 10...».

Стоит также упомянуть о проблеме с файловыми системами. Если Windows установлена в режиме Legacy (BIOS), а Linux в режиме UEFI (или наоборот), они не будут видеть друг друга через стандартные механизмы. В этом случае необходимо привести режимы загрузки обеих систем к общему знаменателю, что часто требует переустановки одной из ОС.

Сценарий Решение Сложность
GRUB обновился и пропала Windows Добавить GRUB_DISABLE_OS_PROBER=false Низкая
Диск заблокирован Windows Отключить Fast Startup в Windows Низкая
Разные режимы загрузки (UEFI/Legacy) Переустановка в едином режиме Высокая

Резюме и заключительные рекомендации

Восстановление видимости Windows 10 в меню GRUB через os-prober является решаемой задачей, требующей внимательного подхода к конфигурации загрузчика и настройкам UEFI. Ключевыми моментами остаются явное включение сканирования в файле /etc/default/grub, отключение функции быстрого запуска в Windows и правильная настройка приоритетов загрузки в BIOS.

Выполнение описанных шагов в большинстве случаев позволяет вернуть полноценную функциональность двойной загрузки без необходимости переустановки операционных систем. Важно помнить о необходимости регулярного обновления загрузчика после любых изменений в конфигурации дисков или установке новых ядер системы.

📌

Проблема решается включением os-prober в конфиге GRUB и отключением Fast Startup в Windows.

Соблюдение осторожности при работе с разделами диска и понимание различий между режимами Legacy и UEFI позволит избежать большинства критических ошибок. Грамотно настроенная система двойной загрузки обеспечивает удобный доступ к ресурсам обеих операционных сред.
